## 1. 产品概述

CRA AI Assistant 是一款面向临床研究领域的桌面工具软件。它利用智谱 AI GLM-4 大语言模型，帮助临床研究助理（CRA）从临床试验方案 PDF、受试者文档和医学影像中自动提取结构化数据，包括入选/排除标准、访视计划、受试者人口统计学信息、用药记录等，并支持 AI 辅助的受试者资格分析，最终一键生成格式化的 Excel 追踪表。

### 产品定位

将 AI 能力嵌入 CRA 日常工作的关键流程中，实现「文档上传 → 智能提取 → 数据管理 → 资格分析 → 报告导出」的端到端自动化，显著减少人工抄写和数据转录的时间。

### 核心价值主张

- **智能而非替代**：AI 完成数据提取的粗加工，用户保留完全的编辑控制权
- **本地化与隐私**：所有文件和数据在本地处理和存储，不上传至任何第三方服务器（仅调用智谱 AI API 进行文本理解）
- **免费模型驱动**：视觉处理使用免费的 GLM-4.6V-Flash 模型，降低使用成本

## 2. 目标用户

### 主要用户

| 用户角色 | 描述 | 核心场景 |
|----------|------|----------|
| **临床研究助理（CRA）** | 负责临床试验现场监查、数据核查和文档管理的研究人员 | 从方案文档中提取入选/排除标准，核对应访视计划，分析受试者资格，生成追踪报告 |
| **临床研究协调员（CRC）** | 在临床试验现场负责受试者管理和数据录入的人员 | 录入和管理受试者访视记录、用药记录，导出数据报告 |

### 用户特征

- 具备临床试验基础知识，了解 GCP（Good Clinical Practice）规范
- 日常工作涉及大量文档阅读和数据转录
- 技术水平参差不齐，需要简单直观的操作界面
- 对数据准确性有极高要求

## 3. 需求背景与痛点分析

### 行业痛点

| 痛点 | 描述 | 影响 |
|------|------|------|
| **人工提取耗时** | CRA 需要逐页阅读几十页甚至上百页的临床试验方案，手动抄录入选/排除标准、访视计划等信息 | 单个方案提取耗时 2-4 小时，且容易遗漏 |
| **数据转录错误** | 手动将文档数据转录到 Excel 追踪表时容易出错 | 数据不一致，监查发现问题后需返工 |
| **格式不统一** | 不同 CRA 的追踪表格式各异，难以汇总和比较 | 增加项目经理审核工作量 |
| **资格判断依赖经验** | 判断受试者是否符合入选/排除标准需要逐条对照方案和受试者资料 | 耗时且容易出现判断偏差 |
| **扫描件处理困难** | 大量临床文档为扫描件或照片，传统工具无法提取文字 | 需要人工逐字录入 |

### 现有解决方案的不足

- **通用 OCR 工具**（如 ABBYY、Tesseract）：仅能提取文字，无法理解临床试验上下文，无法结构化输出
- **人工 Excel 管理**：效率低、格式不统一、容易出错
- **专业 CTMS 系统**（如 Veeva、Medidata）：价格昂贵、部署复杂、学习成本高，不适合中小型研究团队

## 4. 产品目标

### 核心目标

| 指标 | 目标值 | 衡量方式 |
|------|--------|----------|
| **数据提取效率** | 较人工方式提升 80% 以上 | 同一文档提取耗时对比 |
| **数据准确率** | AI 提取准确率 ≥ 85%，经人工修正后达 99% | 抽样比对验证 |
| **资格分析速度** | 单个受试者资格分析 < 30 秒 | 端到端计时 |
| **Excel 导出** | 一键导出标准格式追踪表 | 功能可用性 |

### 约束条件

- 必须为离线可用的桌面应用（仅 AI 推理需要联网）
- 用户无需安装额外依赖（Python、Java 等运行时）
- Windows 10 为主要支持平台，兼容 macOS 和 Linux

## 5. 功能需求

### 5.1 文件管理模块

#### FR-1.1 文件上传

**描述**：用户通过拖拽或点击方式上传临床试验相关文档。

| 需求项 | 详情 |
|--------|------|
| **FR-1.1.1** | 支持拖拽上传和点击选择两种方式 |
| **FR-1.1.2** | 支持批量上传（一次选择多个文件） |
| **FR-1.1.3** | 支持的文件格式：PDF、JPG、JPEG、PNG、BMP、WEBP |
| **FR-1.1.4** | 单文件大小上限默认 50MB，可在设置中调整（1-100MB） |
| **FR-1.1.5** | 自动识别文件类型（PDF / 图片），并展示对应图标 |
| **FR-1.1.6** | 上传后显示文件名、大小、状态标识 |
| **FR-1.1.7** | 文件列表中可单个删除或一键清空 |

#### FR-1.2 文件分区管理

**描述**：文件按用途分为两个独立的存储区域。

| 区域 | 用途 | 上传文件类型 |
|------|------|-------------|
| **方案文件区（Protocol）** | 存放临床试验方案文档 | 包含入选/排除标准、访视计划的方案 PDF |
| **受试者文件区（Subject）** | 存放受试者相关文档 | 受试者访视记录、病历、检查报告等 |

**业务规则**：
- 方案文件和受试者文件分别管理，互不影响
- 方案文件处理时会替换之前提取的标准和访视计划数据（以最新方案为准）
- 受试者文件处理结果为追加模式（支持多个受试者）

#### FR-1.3 文件状态追踪

每个文件具有以下状态流转：

```
待处理 (PENDING) → 处理中 (PROCESSING) → 已完成 (COMPLETED)
                                       → 失败 (FAILED)
```

| 状态 | 颜色标识 | 含义 |
|------|----------|------|
| PENDING | 灰色 | 文件已上传，等待处理 |
| PROCESSING | 蓝色 | AI 正在处理文件 |
| COMPLETED | 绿色 | 处理完成，数据已提取 |
| FAILED | 红色 | 处理失败，显示错误原因 |

### 5.2 AI 数据提取模块

#### FR-2.1 PDF 类型自动检测

**描述**：系统自动判断上传的 PDF 为文本型还是扫描型，并采用不同的处理策略。

| 检测规则 | 处理方式 |
|----------|----------|
| 提取的文字长度 ≥ 100 字符 **且** 有效字符占比 ≥ 30% | 文本型 PDF → 使用 `pdf-parse` 直接提取文字 |
| 不满足上述条件 | 扫描型 PDF → 使用 `node-poppler` 转为 PNG 图片 → GLM-4.6V-Flash 视觉分析 |

**扫描型 PDF 处理细节**：
- 转换分辨率：150 DPI
- 最大转换页数：10 页
- 临时图片存储于系统临时目录，处理完成后自动清理
- 不经过 OCR 步骤，图片直接发送至视觉模型进行理解和分析

#### FR-2.2 方案文件智能提取

**描述**：上传方案文件后，AI 自动并行提取入选/排除标准和访视计划。

**触发方式**：用户在方案文件区上传文件后手动触发处理。

**提取内容**：

| 提取项 | 输出结构 | 说明 |
|--------|----------|------|
| **入选标准** | `[{category, description}]` | 每条标准包含分类和描述 |
| **排除标准** | `[{category, description}]` | 同入选标准结构 |
| **访视计划** | `[{visitType, visitDay, visitWindow, description, items}]` | 包含每次访视的时间窗口和检查项目 |
| **检查项目** | `[{name, category, required}]` | 每个访视下的具体检查项，标注是否必查 |

**处理流程**：

```
方案文件上传
    ↓
PDF 类型检测
    ↓
┌─────────────────┬──────────────────────┐
│  文本型 PDF      │  扫描型 PDF            │
│  pdf-parse 提取  │  node-poppler 转图片   │
│  纯文字          │  GLM-4.6V-Flash 识别   │
└────────┬────────┴──────────┬───────────┘
         ↓                   ↓
      合并文本内容
         ↓
   ┌─────┴─────┐
   │ 并行调用    │
   │ 提取标准   │ ← GLM-4 文本模型
   │ 提取计划   │ ← GLM-4 文本模型
   └─────┬─────┘
         ↓
   结果写入工作表
```

**业务规则**：
- 入选/排除标准和访视计划并行提取，提升处理速度
- 新方案提取结果覆盖旧数据（仅保留最新方案）
- AI 返回的 JSON 经过三层解析策略（直接解析 → Markdown 代码块提取 → 正则匹配）
- 提取的文字内容不足 50 字符时报错，提示用户检查文件

#### FR-2.3 受试者文件智能提取

**描述**：上传受试者文件后，AI 自动提取受试者信息、访视记录和用药信息。

**触发方式**：用户在受试者文件区上传文件后手动触发处理。

**提取内容**：

| 提取项 | 输出结构 | 说明 |
|--------|----------|------|
| **受试者人口统计学** | `{subjectNumber, screeningNumber, randomizationNumber, age, gender, height, weight, ethnicity, birthDate}` | 受试者基本信息 |
| **访视日期** | `[{visitType, date}]` | 各次访视的日期记录 |
| **用药记录** | `[{medicationName, dosage, frequency, route}]` | 药品使用信息 |

**处理流程**：

```
受试者文件上传
    ↓
文件类型判断
    ↓
┌──────────┬───────────────┬──────────────┐
│ 文本 PDF  │ 扫描 PDF       │ 图片文件      │
│ 提取文字  │ 转图片+VLM分析  │ 直接 VLM 分析  │
└────┬─────┴───────┬───────┴──────┬───────┘
     ↓             ↓              ↓
┌─────────┐  ┌───────────────────────┐
│ 文本路径  │  │ 视觉路径               │
│ 串行调用: │  │ extractSubjectData    │
│ · 编号   │  │ FromImage (GLM-4V)    │
│ · 访视   │  └───────────────────────┘
│ · 用药   │
└────┬────┘
     ↓
结果写入工作表（追加模式）
```

**业务规则**：
- 文本型 PDF 串行调用三次 AI（提取编号 → 提取访视 → 提取用药）
- 扫描型/图片文件仅提取人口统计学信息，其余数据需后续单独分析
- 多个受试者的数据追加存储，不覆盖

#### FR-2.4 AI 服务配置

| 参数 | 默认值 | 可调范围 | 说明 |
|------|--------|----------|------|
| API 端点 | `https://open.bigmodel.cn/api/paas/v4/chat/completions` | 任意 URL | 智谱 AI API 地址 |
| 文本模型 | `glm-4` | glm-4 / glm-4.6v-flash | 文本处理使用的模型 |
| 视觉模型 | `glm-4.6v-flash`（固定） | — | 图片处理始终使用免费视觉模型 |
| 处理超时 | 120 秒 | 30-300 秒 | 单次 AI 调用最大等待时间 |
| 重试次数 | 3 次 | 1-5 次 | 网络失败时自动重试 |
| 温度 | 0.1（固定） | — | 低温度确保输出稳定可解析 |
| 最大 Token | 4096（固定） | — | 确保完整返回 |

**重试与容错**：
- 网络错误采用指数退避重试（1s × 重试次数）
- API 参数错误（如 1210）立即返回，不重试，附带详细排查提示
- JSON 解析失败时尝试三种策略（直接解析 → Markdown 提取 → 正则匹配）
- 多文件批量处理时，单个文件失败不影响其余文件继续处理

### 5.4 资格分析模块

#### FR-4.1 多文件资格分析

**描述**：基于已提取的入选/排除标准和受试者文件，AI 自动分析每个受试者是否符合各项标准。

**触发条件**：
- 入选/排除工作表中存在已提取的标准数据
- 受试者文件区中存在已完成处理的文件

**分析流程**：

```
用户点击「分析受试者资格」
    ↓
收集所有已完成处理的受试者文件路径 + 当前标准数据
    ↓
逐文件处理：
    ↓
┌──────────────┬───────────────────┐
│ 文本型 PDF    │ 扫描型/图片         │
│ analyze       │ analyzeEligibility │
│ Eligibility   │ FromImage          │
│ (GLM-4)       │ (GLM-4.6V-Flash)  │
└──────┬───────┴────────┬──────────┘
       ↓                ↓
   AI 返回 {inclusion: [{id, eligible, reason}], exclusion: [...]}
       ↓
   ID 规范化（三层匹配）
       ↓
   按标准 ID 聚合多文件结果
       ↓
   写入 fileResults 数组
```

#### FR-4.2 AI 返回 ID 规范化

AI 返回的标准 ID 可能与前端 ID 不匹配，系统采用三层匹配策略：

| 优先级 | 匹配策略 | 示例 |
|--------|----------|------|
| 1 | 直接 ID 匹配（含清理括号/前缀后匹配） | `[ID: abc123]` → `abc123` |
| 2 | 数字索引匹配（AI 返回序号转为实际 ID） | `1` → 第一条标准的 ID |
| 3 | 描述相似度匹配（Levenshtein 距离，阈值 0.85） | AI 返回的描述文本与标准描述比较 |

#### FR-4.3 多文件结果合并展示

当存在多个受试者文件时，每条标准显示合并后的单一结论：

| 情况 | 展示逻辑 | 示例 |
|------|----------|------|
| 单个文件 | 直接显示该文件结果 | 「符合：受试者年龄 45 岁，在 18-65 岁范围内」 |
| 所有文件一致 | 「所有文件均符合/不符合」 | 「所有文件均符合 (3/3)」 |
| 文件结果不一致 | 多数投票法 | 「多数符合 (2/3 个文件)」 |

**业务规则**：
- 资格分析结果不跨会话持久化，每次启动应用时自动清除（避免过时结果误导用户）
- 保留向后兼容的单文件字段（`eligible` + `reason`）
- 单个文件分析失败不影响其余文件继续处理

### 5.5 Excel 导出模块

#### FR-5.1 一键导出追踪表

**描述**：将所有工作表数据导出为格式化的 Excel 文件。

**触发方式**：点击顶部导航栏的「导出 Excel」按钮。

**前置条件**：至少一个工作表有数据（否则按钮禁用）。

**导出内容**：

| 工作表 | 列定义 | 数据来源 |
|--------|--------|----------|
| **入选标准** | 序号、分类、标准描述 | InclusionCriteria[] |
| **排除标准** | 序号、分类、标准描述 | ExclusionCriteria[] |
| **访视计划** | 访视名称、访视时间、时间窗口、描述、检查项目 | VisitSchedule[] |
| **受试者访视** | 受试者编号、访视名称、计划日期、实际日期、状态、备注 | SubjectVisitData[] |
| **用药记录** | 受试者编号、访视、药品名称、剂量、频率、给药途径、开始日期、结束日期、备注 | MedicationRecord[] |
| **受试者信息** | 受试者编号、年龄、性别、身高(cm)、体重(kg)、民族、出生日期、筛选号、随机号 | SubjectDemographics[] |

**格式规范**：

| 要素 | 规格 |
|------|------|
| 字体 | 微软雅黑 |
| 标题行 | 合并单元格，18pt 加粗 |
| 表头行 | 白色文字蓝色背景（#4472C4），11pt 加粗，冻结 |
| 数据行 | 隔行交替样式 |
| 日期格式 | yyyy-mm-dd |
| 默认文件名 | `临床试验追踪表_{日期}.xlsx` |
| 空工作表 | 默认跳过（无数据的工作表不创建） |

**保存流程**：
1. 弹出系统原生保存文件对话框
2. 默认路径为系统下载目录
3. 写入完成后弹出成功提示，显示文件路径

## 6. 用户流程

```
启动应用
    ↓
配置 API Key（预置默认 Key，可跳过）
    ↓

    ↓

### 6.2 方案处理主流程

```
上传方案文件（拖拽/点击）
    ↓
触发文件处理
    ↓
AI 自动提取：入选标准 + 排除标准 + 访视计划
    ↓

    ↓

    ↓
AI 自动提取：受试者信息 + 访视记录 + 用药记录
    ↓
查看并编辑受试者数据
    ↓
（可选）执行资格分析
    ↓
导出 Excel 追踪表
```

```

    ↓

    ↓

    ↓

    ↓

## 7. 数据模型

### 7.1 核心实体关系

```
┌─────────────┐     ┌──────────────┐
│  FileInfo    │     │ AppSettings  │
│  (文件信息)   │     │  (应用设置)   │
└─────────────┘     └──────────────┘

┌─────────────────┐        ┌─────────────────┐
│ InclusionCriteria│        │ ExclusionCriteria│
│  (入选标准)       │        │  (排除标准)       │
│                 │        │                 │
│ fileResults[] ──┼── 1:N ─┼→ InclusionFileResult
│                 │        │  ExclusionFileResult
└─────────────────┘        └─────────────────┘

┌─────────────────┐
│  VisitSchedule   │
│  (访视计划)       │
│                  │
│  items[] ── 1:N ─┼→ VisitItem (检查项目)
└─────────────────┘

┌─────────────────┐
│ SubjectVisitData │
│  (受试者访视)     │
│                  │
│  visits[] ─ 1:N ─┼→ SubjectVisitItemData
│                  │     └→ items[] → VisitItemDetail
└─────────────────┘

┌─────────────────┐     ┌──────────────────┐
│ MedicationRecord │     │ SubjectDemographics│
│  (用药记录)       │     │  (人口统计学)      │
└─────────────────┘     └──────────────────┘
```

### 7.2 持久化策略

| 数据类别 | 是否持久化 | 存储位置 | 说明 |
|----------|-----------|----------|------|
| 应用设置 | 是 | `{userData}/settings.json` | 独立文件存储 |
| 入选/排除标准 | 是 | Zustand persist (localStorage) | 包含标准内容，不含资格结果 |
| 访视计划 | 是 | Zustand persist | — |
| 受试者访视 | 是 | Zustand persist | — |
| 用药记录 | 是 | Zustand persist | — |
| 受试者人口统计学 | 是 | Zustand persist | — |
| 资格分析结果 | **否** | 内存 | 每次启动自动清除 |
| 文件列表 | **否** | 内存 | 文件路径在会话间不保留 |
| 处理状态 | **否** | 内存 | — |

## 8. 非功能需求

### 8.1 安全性

| 需求 | 实现方式 |
|------|----------|
| 进程隔离 | `nodeIntegration: false`、`contextIsolation: true`、`sandbox: true` |
| 通信安全 | Preload Script 通过 `contextBridge` 暴露有限 API |
| CSP 策略 | 生产环境仅允许 `self` + `https://open.bigmodel.cn` |
| API Key 保护 | 不在日志中输出，设置界面使用密码输入框 |
| 文件路径验证 | 主进程验证所有来自渲染进程的文件路径参数 |

### 8.2 性能

| 指标 | 要求 |
|------|------|
| 应用启动时间 | < 5 秒（冷启动） |
| 文件上传响应 | 即时反馈，上传无感知延迟 |
| AI 调用超时 | 默认 120 秒，可配置 30-300 秒 |
| PDF 内容缓存 | 同一文件多次分析时复用已提取内容 |
| PDFProcessor 延迟加载 | 动态 `import()` 避免 pdfjs-dist 在启动时加载 |

### 8.3 可靠性

| 需求 | 实现方式 |
|------|----------|
| AI 调用重试 | 网络错误自动重试（指数退避），最多 3 次 |
| 多文件容错 | 单个文件分析失败不阻断批量处理 |
| JSON 解析容错 | 三层解析策略（直接 → Markdown → 正则） |
| 临时文件清理 | 图片转换后自动删除，ENOENT 错误静默处理 |
| ID 匹配容错 | 三层 ID 规范化，最大程度匹配 AI 返回结果 |
| 异常兜底 | `uncaughtException` 和 `unhandledRejection` 全局捕获 |

### 8.4 兼容性

| 平台 | 要求 |
|------|------|
| Windows | 10+ (x64)，NSIS 安装包 |
| macOS | 10.15+，DMG 安装包 |
| Linux | AppImage 格式 |
| 依赖打包 | 所有运行时依赖（含 Poppler 二进制）打包在安装包内，用户无需额外安装 |

### 8.5 可用性

| 需求 | 说明 |
|------|------|
| 界面语言 | 简体中文 |
| 错误提示 | 所有错误信息使用中文，含具体排查建议 |
| 空状态引导 | 工作表无数据时显示引导提示 |
| 处理状态反馈 | 顶部显示加载动画，文件状态实时更新 |
| API 状态指示 | 顶部导航栏显示 API 连接状态（已连接 / 未配置） |
